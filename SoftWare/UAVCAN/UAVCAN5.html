1、Cyphal传输层概述

实际上Cyphal底层的传输协议不一定是can，只不过最初是基于can总线。

前面说过Cyphal提供了两种传输方式，广播消息和服务请求。

前面也说过，Cyphal传输的消息以IDL形式描述，

先给大家看一眼IDL描述的协议：

uint16[<=3] foo
int32 bar
@sealed

将上述的协议保存成xxx.uavcan文件，使用Cyphal提供的编译工具编译后会生成一系列的数据结构，包括：一个包含上述字段的结构体，序列化和反序列化该结构体的两个函数等。

消息经过序列化之后，接收端如何进行反序列化呢？将一条消息进行反序列化，总需要知道它序列化之前的格式吧？所以Cyphal在传送消息时，除了发送消息序列化之后的数据（payload）之外，在传输层还会附上该消息的data type id，该id必须是唯一，这样接收端就可以根据该id来辨别这是一条什么消息，从而进行反序列化。

传输层事实上除了附上data type id之外，还会附上以下信息：

对于广播消息和服务消息，附加信息略有不同：

  • 广播消息

    **对于广播消息，ID最大是8191。**也就是支持8192条广播。

  • 服务消息

服务消息分请求和响应，唯一的区别就是源node id和目的node id含义相反。

（1）请求：源 node id是指客户端，因为请求自然是由客户端发起的。


（2）响应：响应自然是服务器回应的，所以源node id指的是服务器node id。

**对于服务消息，消息ID最大是511。**也就是支持512条请求。

客户端和服务器的node id在通信网络里必须是唯一的。

下面对各个字段进行解释：

  • node id为0的广播消息叫匿名消息。

  • 消息比较短，单个can帧就可以携带，这种传输叫单帧传输；
    消息较长，需要几个can帧传输，这种叫多帧传输。
    transfer id这个用于多帧传输的情况，标识这是同一次传输，即一个消息由于过长被拆分成若干can帧时，这些can帧的传输id都相同，组装的时候利用该规则进行组装。一次传输结束，tranfer id递增，表示开始下一次会话。（有点类似于IP协议的分片和重组）

  • payload就是序列化后的消息数据。

  • priority指消息的优先级，优先级高的将优先获取总线占用权。

那么Cyphal描述的这些信息是如何和can帧对应的？换句话说，can帧如何承载Cyphal消息。

2、can帧如何承载Cyphal消息

2.1 29位的can id布局

Cyphal采用can2.0 B协议，即扩展帧，29位标识符。

Cyphal基于can协议之上，封装了一层，Cyphal相当于处于传输层和应用层。

标准帧和扩展can帧的格式如下：

Cyphal利用了can协议数据帧的仲裁段的ID字段，将ID字段拆分成了若干字段，赋予了每一个字段新的含义。我们也可以看到，对于扩展帧，它的29位ID在空间上并非连续，被分割成两部分，所以在解析时需要先提取出ID，再将ID解析成Cyphal定义的若干字段。

Cyphal只是对CAN帧的ID字段和数据段做了一些重新设计。除此之外并无其他改动，所以严格来讲，Cyphal并不能说位于can协议之上，因为它并没有像TCP协议相对于IP协议，增加了一个TCP报头，整个IP协议包作为TCP协议的报文数据。

我们先看广播消息。

广播消息can id的低7位用于存放发布消息的节点id，也就是这条消息是谁发送的。

第八位是保留位，填0。

中间13位用于存储消息id。这决定了消息id的范围是0-8191。

2.2 Cyphal payload的构成

Cyphal对CAN帧数据字段的修改

1、单帧结构

单帧最大8个字节，其中数据是可变的，最大占7个字节，最后一个字节叫尾部字节。

尾部1字节8个位拆分成了几个字段，用于携带协议信息。

Field Bits Description
Start of transfer 1 标识该帧是此次传输的第一帧，对于单帧传输，该位自然总是1
End of transfer 1 标识该帧是此次传输的最后一帧，对于单帧传输，该位自然总是1
Toggle bit 1 对于单帧传输，该位总是0，该位是给多帧传输用的，对于第一帧该位为0，下一帧为1，如此翻转下去，用途是检测和避免重复can帧的错误
Transfer ID 5 标识此次传输的id，0-31循环覆盖

2、多帧结构

多帧传输除了第一帧与单帧不同之外，后面每一帧的结构和单帧一样。第一帧多了两个字节的CRC校验值，该校验是对所有帧的校验。
